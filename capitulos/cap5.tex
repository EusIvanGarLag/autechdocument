\section{Arquitectura del Sistema}
\label{sec:arquitectura}
% ============================================================

AUTECH es una aplicación móvil desarrollada con Flutter que sigue una separación de
responsabilidades bastante clara, aunque sin aplicar textualmente un \textit{framework}
arquitectónico de libro. La estructura surgió de la manera más natural, ya que, conforme el
proyecto fue creciendo: primero había una sola pantalla de \textit{login}, después vinieron los
módulos de simulación, y para cuando se integró Supabase la lógica de negocio ya vivía en su
propio archivo. Eso, si lo vemos desde otro punto de vista, es exactamente lo que propone la
\textit{Layered Architecture} \cite{martin2012}: separar el sistema en capas que apuntan a capas
adyacentes, y dentro de las mismas se establecen las reglas del sistema y cómo conviven los
componentes entre su misma capa y capas adyacentes.

La arquitectura se divide en cinco capas principales que se corresponden con la presentación,
el dominio y los datos. Su organización de directorios no es tan estricta como en proyectos
empresariales, pero la intención es la misma. Se detalla la función de cada nivel arquitectónico,
resaltando cómo la integración con Supabase y el enfoque \textit{offline-first} soportan las
acciones que se hacen desde la aplicación AUTECH:

\begin{itemize}
    \item \textbf{Capa de Presentación (\textit{Presentation Layer}):} Contiene los componentes
    visuales desarrollados en Flutter, organizados en módulos como autenticación (Auth~·~Splash),
    paneles de usuario (Administrador, Profesor, Alumno) y simuladores. Esta capa sigue un enfoque
    reactivo mediante el uso de \texttt{StreamBuilder} y \texttt{FutureBuilder}, lo que permite
    actualizar la interfaz en tiempo real conforme cambian los datos. Su responsabilidad es
    estrictamente visual, dejando toda la lógica de negocio a la capa de servicios, lo que
    garantiza una clara separación de responsabilidades y facilita la mantenibilidad.

    \item \textbf{Capa de Servicios (\textit{Service Layer}):} Actúa como intermediaria entre la
    interfaz de usuario y las fuentes de datos, abarcando la lógica de aplicación mediante
    servicios como \texttt{AuthService}, \texttt{UserService}, \texttt{GrupoService},
    \texttt{EjercicioService} y \texttt{SupabaseService}. Además, integra componentes clave como
    \texttt{OfflineSyncService}, que habilita el enfoque \textit{offline-first} al gestionar la
    sincronización entre almacenamiento local y remoto, y \texttt{ApiService}, encargado de la
    comunicación con el \textit{backend} externo. Esta capa implementa patrones como
    \textit{Singleton} y \textit{Facade}, centralizando la lógica y evitando la duplicidad de
    código.

    \item \textbf{Capa de Datos (\textit{Data Layer}):} Gestiona la persistencia de la información
    tanto en entornos locales como remotos. Incluye bases de datos locales (SQLite) y
    almacenamiento de recursos (Assets JSON) para garantizar la disponibilidad sin conexión, así
    como servicios remotos de Supabase (PostgreSQL, GoTrue Auth y Realtime). Esta capa abstrae el
    acceso a los datos, permitiendo que la aplicación opere de manera eficiente y resiliente ante
    fallos de conectividad, priorizando siempre el acceso local y sincronizando posteriormente con
    la nube.

    \item \textbf{Backend Externo (Infraestructura de Procesamiento):} Corresponde a una API REST
    desplegada en Render, encargada de ejecutar procesos especializados relacionados con teoría de
    la computación, como la simulación de autómatas (AFD, AFN, APD, MT) y operaciones como
    conversión (AFN~$\rightarrow$~AFD), minimización (MIN) y otras operaciones (OPR). Esta capa
    permite descargar el procesamiento intensivo del dispositivo cliente, manteniendo la aplicación
    ligera y escalable.

    \item \textbf{Gestión de Autenticación y Seguridad:} La autenticación se gestiona de forma
    centralizada a través de Supabase Auth, accedida mediante el \texttt{AuthService}. Este
    servicio valida credenciales, gestiona sesiones y controla el acceso a los diferentes módulos
    según el rol del usuario (Administrador, Profesor o Alumno). Aunque la aplicación implementa
    un enfoque \textit{offline-first} para los datos, la autenticación se realiza exclusivamente
    en línea, garantizando la seguridad y consistencia del sistema.
\end{itemize}

\subsubsection*{Flujo de Control General}

El punto de entrada es \texttt{main.dart}, que inicializa Supabase y el \textit{factory} de SQLite
(\texttt{sqflite\_common\_ffi} en \textit{desktop}) antes de lanzar la aplicación con
\texttt{SplashScreen} como pantalla inicial. La \texttt{SplashScreen} verifica si hay sesión activa
con Supabase Auth (GoTrue) y, en caso afirmativo, consulta el registro de usuario en PostgreSQL para
obtener el rol. A partir de ahí, la navegación se bifurca según ese rol.

Dentro de cada módulo de simulación (\texttt{RegexScreen}, \texttt{PdaScreen},
\texttt{TuringScreen}) el flujo sigue siempre el mismo patrón: el usuario configura el autómata en
un formulario o en el \textit{canvas} interactivo (esto depende de la opción que está siendo
ocupada); al presionar el botón de simular, \texttt{ApiService} serializa la definición a JSON y
hace el POST al \textit{endpoint} correspondiente, y el resultado se deserializa para actualizar la
visualización en \texttt{AutomatonCanvas} paso a paso.

\section{Diseño de Datos}
\label{sec:diseno_datos}
% ============================================================

La base de datos del sistema es PostgreSQL gestionada a través de Supabase, una base de datos
relacional con soporte completo de esquemas, llaves foráneas e integridad referencial. A diferencia
del modelo NoSQL anterior (Cloud Firestore), aquí las relaciones entre entidades se expresan
mediante \textit{Foreign Keys} explícitas y tablas de unión normalizadas. Supabase añade sobre
PostgreSQL un sistema de \textit{Row Level Security} (RLS) que permite aplicar políticas de acceso
directamente en la base de datos, sin depender únicamente de la lógica de la aplicación cliente
\cite{google2024b}\cite{supabase2024}.

Esta decisión de diseño tuvo consecuencias prácticas importantes durante el desarrollo. Por ejemplo,
las preguntas de un ejercicio están embebidas en el documento del ejercicio como columnas JSONB en
la tabla de \texttt{group\_members}, lo cual simplifica las lecturas de progreso (una sola consulta
por alumno) pero requiere funciones RPC en el servidor para actualizaciones parciales seguras. Es el
\textit{trade-off} clásico entre normalización y rendimiento de lectura: se prefirió JSONB para el
progreso por nodo porque su estructura es semiestructurada y varía por unidad \cite{fowler2013}.

\subsection{Modelo Entidad-Relación (PostgreSQL / Supabase)}
\label{subsec:mer}

El esquema lógico de PostgreSQL se organiza en seis tablas principales. A diferencia de Firestore,
PostgreSQL impone tipos de columna y restricciones en tiempo de escritura \cite{google2024b}, por lo
que el control de tipos se divide entre la base de datos (constraints, CHECK, NOT NULL) y la
aplicación cliente a través de los métodos \texttt{fromMap()}/\texttt{toMap()} de cada modelo en
Dart. \textit{Row Level Security} (RLS) refuerza el control de acceso a nivel de fila sin
necesidad de lógica adicional en el cliente.

\section{Ingeniería del Motor de Simulación}
\label{sec:motor_simulacion}
% ============================================================

El motor de simulación es la parte técnicamente más interesante del proyecto. Está dividido en dos
componentes que trabajan juntos: un \textit{widget} de \textit{canvas} interactivo que corre
completamente en el cliente (\texttt{AutomatonCanvas}), y una API REST en Python que implementa los
algoritmos de teoría de autómatas. Todo el procesamiento formal de cadenas, las transformaciones
entre representaciones y la verificación de propiedades de los lenguajes se apoyan en resultados
clásicos de la teoría de lenguajes formales y autómatas \cite{aho1995}.

La decisión de separar así las responsabilidades tuvo que ver con el rendimiento y con la
complejidad algorítmica. Implementar en Dart la conversión de NFA a DFA por subconjuntos, la
minimización de Hopcroft o la eliminación de estados para obtener una expresión regular hubiera
sido técnicamente posible, pero bastante más código a mantener. Sin embargo, algunos de esos
algoritmos sí terminaron implementándose en el cliente para el modo \textit{offline} del módulo de
Expresiones Regulares.

transformaciones de zoom y paneo controladas mediante \texttt{TransformationController}.

La relación entre \texttt{AutomatonCanvas} y los \textit{screens} que lo contienen sigue el patrón
de \textit{widget} controlado: el \textit{canvas} no mantiene estado propio de los nodos, solo de
sus posiciones y del modo de edición. Los datos reales están dentro del \texttt{State} del
\textit{screen} padre y se pasan al \textit{canvas} mediante \textit{props}. Esto facilita mucho
la sincronización cuando el \textit{backend} devuelve un autómata nuevo: simplemente se reemplaza
la lista y Flutter reconstruye el \textit{canvas} \cite{flutter2024c}.

    (1 segundo).
    \item \textbf{Latencia P95 (0.725~s):} El 95\% de las peticiones se completaron en menos de
    725~ms.
    \item \textbf{Desviación Estándar (0.175~s):} Confirma la estabilidad de la conexión.

\subsection{Contexto del Análisis Comparativo}
\label{subsec:contexto_comparativo}

El desarrollo de AUTECH surge de una necesidad pedagógica identificada en el contexto de la materia
de Teoría de la Computación de la Escuela Superior de Cómputo del IPN \cite{ipn2020}\cite{escom2016}.
Para evaluar en qué medida AUTECH representa una aportación genuina, es necesario contrastarla con
las herramientas existentes más relevantes: \textbf{JFLAP} \cite{rodger2006}\cite{sipser2012},
\textbf{Automata Tutor v3} \cite{aichernig2022}, \textbf{Regex101} \cite{fycikas2021} y
\textbf{RegExr} \cite{skinner2021}, y \textbf{FSAM}. El análisis cubre tres dimensiones:
algoritmos implementados, experiencia pedagógica y accesibilidad técnica.

\subsection{Diferencias y Mejoras Algorítmicas}
\label{subsec:diferencias_algoritmicas}

\subsubsection*{Expresión regular a AFN-$\varepsilon$ (Construcción de Thompson)}

AUTECH implementa un \textbf{parser Shunting-Yard de Dijkstra} \cite{aho1995} para convertir la
expresión regular a notación postfija, normalizando previamente los operadores $a+ \rightarrow aa^*$
y $a? \rightarrow (a|\varepsilon)$, e insertando operadores de concatenación implícita. Sobre la
forma postfija se construye el AFN-$\varepsilon$ mediante Thompson \cite{thompson1968}.

\subsubsection*{$\varepsilon$-clausura y AFN-$\varepsilon$ a AFD}

El cálculo de la $\varepsilon$-clausura usa un \textbf{algoritmo DFS iterativo con pila explícita}
(no recursivo). La construcción de subconjuntos usa \textbf{BFS con poda integrada de estados
inalcanzables}: solo se generan los subconjuntos accesibles desde el estado inicial
\cite{hopcroft2007}.

\subsubsection*{Minimización de AFD: Tabla de distinguibilidad + Union-Find}

AUTECH añade una \textbf{estructura Union-Find con \textit{path compression}} para fusionar
eficientemente los estados indistinguibles \cite{hopcroft2007}. Además, \textbf{elimina
automáticamente el estado muerto resultante} si no tiene transiciones útiles,
presentando solo los estados semánticamente significativos.

\subsubsection*{AFD/AFN a Expresión Regular: GNFA con simplificación algebraica}

AUTECH aplica el mismo esquema GNFA de JFLAP, pero añade una etapa de
\textbf{post-procesamiento algebraico mediante un AST de expresión regular} que simplifica en hasta
doce iteraciones, incluyendo \textbf{factorización de prefijos comunes
($ab|ac \rightarrow a(b|c)$)} \cite{aho2006}.

\subsubsection*{Simulación de APD: BFS con terminación garantizada}

AUTECH implementa la simulación del APD mediante \textbf{BFS con conjunto de configuraciones
visitadas}, garantizando terminación y completitud. Un límite de 200 símbolos en pila previene
desbordamientos de memoria, protección ausente en JFLAP \cite{rodger2006}\cite{hopcroft2007}.

\subsubsection*{Simulación de MT: Traza \textit{batch} completa}

AUTECH \textbf{pre-computa la traza completa de hasta 1\,000 pasos en el servidor Python} y la
entrega al cliente en una sola respuesta JSON \cite{fielding2000}. Cada paso registra la
configuración completa en notación estándar $uqv$ \cite{sipser2012}.

\subsection{Ventajas Pedagógicas Respecto a las Herramientas Existentes}
\label{subsec:ventajas_pedagogicas}

\begin{itemize}
    \item \textbf{Frente a JFLAP \cite{rodger2006}:} AUTECH presenta cada operación con
    visualización dinámica paso a paso y código de color por componente, reduciendo la carga
    cognitiva \cite{sweller1988} asociada a seguir manualmente la tabla de transiciones de JFLAP.

    \item \textbf{Frente a Automata Tutor v3 \cite{aichernig2022}:} mientras Automata Tutor evalúa
    si el autómata del estudiante es correcto, AUTECH enseña el proceso de construcción paso a
    paso, combinando evaluación con retroalimentación inmediata.

    \item \textbf{Frente a FSAM:} AUTECH integra todos los modelos (AFD, AFN, APD, MT) en un
    sistema unificado con soporte \textit{offline} \cite{supabase2024}, funcionalidad sin
    equivalente en FSAM.

    \item \textbf{Frente a Regex101 y RegExr \cite{fycikas2021}\cite{skinner2021}:} AUTECH
    construye el AFN-$\varepsilon$ correspondiente y lo simula, conectando la expresión regular con
    la teoría formal \cite{hopcroft2007}\cite{thompson1968}.

    \item \textbf{Ruta de aprendizaje estructurada:} AUTECH integra la simulación dentro de una
    ruta de contenido con ocho unidades progresivas, ejercicios de teoría, práctica y
    miniexámenes, racha de actividad, medallas de logro y seguimiento por grupo escolar, aplicando
    los principios de gamificación educativa \cite{deterding2011} y aprendizaje activo
    \cite{felder2016}.
\end{itemize}

\subsection{Ventajas Técnicas y de Accesibilidad}
\label{subsec:ventajas_tecnicas}

Ninguna de las herramientas de referencia ofrece una aplicación móvil nativa. AUTECH, construida
con Flutter \cite{flutter2024c}, ejecuta nativamente en Android, iOS, Windows, macOS y Linux desde
una única base de código, con rendimiento nativo en cada plataforma gracias al motor de renderizado
Impeller \cite{flutter2024a}. Los algoritmos de AFD, AFN y expresiones regulares se procesan
completamente en el cliente sin latencia de red \cite{fielding2000}, mientras que los modelos más
complejos (APD, MT) se delegan a una API REST en Python.

Además, AUTECH implementa un \textbf{sistema de sincronización \textit{offline}} con cola de
operaciones en SQLite local y validación de \textit{timestamps} anti-trampa \cite{supabase2024},
sin equivalente en ninguna de las herramientas de referencia.

\newpage
\subsection{Conclusión General -- Ciclo 0}
\label{subsec:conclusion_general}

El análisis comparativo demuestra que AUTECH no es simplemente una replicación de las herramientas
existentes en una plataforma móvil. Las diferencias son cualitativas en tres niveles:

\textbf{Nivel algorítmico:} AUTECH introduce mejoras concretas sobre JFLAP en seis de los ocho
algoritmos evaluados: simplificación post-GNFA, Union-Find en minimización, BFS para APD, cuatro
etapas de limpieza en la conversión APD a GIC, traza \textit{batch} para MT y poda integrada en la
construcción de subconjuntos.

\textbf{Nivel pedagógico:} La integración de la simulación dentro de una ruta de aprendizaje
estructurada con retroalimentación inmediata, gamificación y seguimiento de progreso por grupo
escolar no tiene equivalente en ninguna de las herramientas analizadas
\cite{deterding2011}\cite{felder2016}\cite{sweller1988}.

\textbf{Nivel de accesibilidad:} La disponibilidad multiplataforma incluyendo dispositivos móviles,
el modo \textit{offline} funcional y la eliminación de dependencias externas eliminan barreras de
acceso que limitan el uso práctico de las herramientas existentes en el contexto universitario
mexicano \cite{crompton2018}\cite{kukulska2007}.
